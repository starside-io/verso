Overview

A Verso project is a folder. The folder holds a deck.json manifest, a slides/ directory of one JSON file per slide, an optional themes/ directory of project themes, and any assets you reference (images, fonts).

The CLI runs three roles against that folder: a viewer (verso dev), a visual editor (verso edit), and a static builder (verso build). All three read the same JSON. None of them stores hidden state.

my-deck/
├── deck.json
├── slides/
│   ├── 01-cover.json
│   └── 02-pricing.json
├── themes/
│   └── my-brand.json     # optional
└── assets/
    └── logo.svg

CLI commands

CommandWhat it does
verso init <dir> [--template <t>]Scaffold a deck. Templates: minimal, layouts-gallery, multi-path, branded, inline-theme, extended.
verso devStart the viewer with file-watch. Default port 5173.
verso editStart the visual editor + viewer side by side. Default editor 5180, viewer 5173.
verso new slide <id>Create a stub slide and append it to slide_order.
verso new slide --from <file.md>Import a Markdown outline. Each # heading becomes a slide; nested lists become bullets; fenced code becomes code blocks.
verso new layout <id>Scaffold a custom layout in layouts/.
verso theme <name>Set the active theme in the manifest.
verso build [--format pdf|html|png] [--path <id>]Render the deck to disk. Default pdf. --no-inline-images for HTML keeps URLs instead of base64-inlining.

Run any command with --help for full flags.

Deck manifest

deck.json is the spine. It declares the deck title, the active theme, every audience path, the slide order, and any deck-wide variables, watermark, or style overrides.

{
  "title": "Q4 2026 readout",
  "theme": "verso-slate",
  "paths": {
    "sales": { "label": "Sales", "color": "#5eead4" },
    "engineering": { "label": "Engineering", "color": "#a78bfa" }
  },
  "slide_order": ["01-cover", "02-context", "03-arch", "04-closing"],
  "variables": { "author": "M. Burlet" },
  "watermark": { "text": "DRAFT", "position": "bottom-right", "opacity": 0.18 },
  "style_overrides": { "primary": "#1A3C6E" }
}

Section markers

slide_order can also include section objects: { "id": "part-2", "title": "Part 2: Architecture" }. Sections appear in the auto-built agenda and as dividers in the editor's slide list.

Slide schema

One file per slide in slides/. Filename is up to you; id in the JSON is the canonical handle.

{
  "id": "04-pricing",
  "title": "Pricing",
  "header": "Q4 2026",
  "layout": "two-col",
  "transition": "slide-left",
  "align": { "horizontal": "center", "vertical": "middle" },
  "path_include": ["sales", "exec"],
  "omit_from_agenda": false,
  "notes": "Hammer the cost-of-ownership story.",
  "annotation": "Internal review v3",
  "style_overrides": { "accent": "#fb923c" },
  "content": [ /* blocks */ ]
}

Required vs optional

  • Required: id, layout.
  • Optional: everything else. Most slides will at least set title and content[].

Layouts (17 built-in)

The layout decides where blocks go on the slide. Pick one with "layout": "...".

Core

  • content · single column, centered
  • two-col · splits content evenly into two columns
  • three-col · three even columns
  • image-left / image-right · image on one side, blocks on the other
  • hero · oversized title and subtitle
  • full-image · edge-to-edge image with optional caption

Openers & closers

  • cover · title slide, kicker line, footer
  • section · large section divider
  • closing · "thank you" / next-steps
  • author · presenter card with avatar and bio

Structured

  • agenda · auto-built from the deck outline (or manual bullets). omit_from_agenda: true hides the slide from its own table of contents.
  • compare · side-by-side comparison columns
  • stats · row of metric cards
  • big-number · single oversized stat with a label
  • quote · pulled quote with attribution
  • timeline · ordered milestones

Content blocks

Every entry in content[] is a tagged object. Blocks can carry path_include, path_exclude, and style_overrides.

Text

  • heading · { text, level: 1-4 }
  • text · { text } (paragraph)
  • bullets · { items: string[] }
  • quote · { text, attribution? }

Media

  • image · { src, alt?, caption? }
  • code · { source, language? }
  • embed · { src, title?, aspect?, allow?, fallback_src?, fallback_text? }. PDF / print swaps in the fallback because headless Chromium doesn't reliably rasterize cross-origin frames.

Containers (recursive)

  • card · { tone?, variant?, content: ContentBlock[] }
  • panel · { tone?, variant?, content: ContentBlock[] }

Decoration

  • callout · { text, tone? }
  • badge · { text, tone?, variant? }
  • accent-bar / divider · visual rules
  • custom · for project-defined components, looked up via the runtime registry

Tones: primary · secondary · accent · muted · surface. Variants: soft · solid · outline.

Themes & the color cascade

A theme is a small JSON file (built-in or in themes/) that names three semantic color roles. Layouts only ever reference CSS variables, never hex.

{
  "name": "my-brand",
  "colors": {
    "primary": "#1A3C6E",
    "secondary": "#4A90D9",
    "classic": "#F4F4F4",
    "background": "#0a0a14",
    "foreground": "#edeef5"
  }
}

Built-in themes

  • verso-slate (default), verso-warm, verso-mono, verso-neon, verso-mars, verso-forest.

Override cascade

Theme defaults → deck style_overrides → slide style_overrides → block style_overrides. Each level emits CSS variables and only affects its own scope. accent falls back to secondary; surface and muted are derived from primary/secondary via color-mix() if not explicitly set.

Path branching

The headline feature. Paths are declared in the manifest and referenced per slide and per block.

{
  "layout": "two-col",
  "content": [
    /* shared block, no path filter */
    { "type": "heading", "text": "Pricing" },
    /* sales-only block */
    { "type": "bullets", "path_include": ["sales"], "items": [ /* … */ ] },
    /* exec-only block */
    { "type": "bullets", "path_include": ["exec"], "items": [ /* … */ ] }
  ]
}

Resolution rules

  • If path_include is set, the slide/block only renders for paths in that list.
  • If path_exclude is set, those paths are removed from the inclusion set.
  • If neither is set, the slide/block renders in every path.
  • The active path is read from ?path=<id>. With no ?path= the viewer shows a path picker.

Variables

Any text field can reference {{key}}. Resolved at render time after HTML escaping.

Built-in keys

  • {{date}} · today as YYYY-MM-DD
  • {{time}} · render time as HH:MM
  • {{deckTitle}} · manifest.title
  • {{pathId}} · the active path id

Custom keys

Declare in manifest.variables (or via the editor's ⚙ Deck dialog). Custom keys win on collision with built-ins.

Caveat: substitution runs on the rendered HTML, so {{key}} inside a code block also gets substituted. Escape with a different sigil if you need literal braces in code samples.

Transitions (16 built-in)

Set slide.transition to opt in. Plays when the slide becomes active in present or speaker mode. Skipped under prefers-reduced-motion: reduce.

  • Off: none (default)
  • Move: fade, slide-left, slide-right, slide-up, slide-down
  • Scale: zoom, zoom-out, pop
  • 3D: flip-x, flip-y, tilt
  • Reveal: iris, wipe-right, wipe-down
  • FX: blur

The field is a free-form string at the schema level: register your own .verso-transition-<name> keyframes in styles.css and set the name on the slide.

Visual editor

verso edit opens a three-pane editor: slide list (left), live preview (center), inspector / transitions / themes (right). A JSON panel at the bottom shows the active slide with syntax highlighting.

Toolbar

  • + Slide, + Block ▾ (categorized), Layout ▾ (17 built-ins), Align ▾
  • ⚙ Deck · edit deck title and custom variables
  • Watermark · text, position, opacity
  • ⌕ Find · deck-wide find & replace (Cmd/Ctrl+Shift+F); regex + case-sensitive flags
  • ↶ ↷ · per-slide undo/redo, 50-step cap
  • ▶ Present · opens the active path in present mode
  • ⤓ Export ▾ · PDF (default), HTML, PNG (per slide)

Hotkeys

  • Cmd/Ctrl+K · slide search palette
  • Cmd/Ctrl+Shift+F · find & replace
  • Cmd/Ctrl+Z / Cmd/Ctrl+Shift+Z · undo / redo

Image upload

Drag a file onto an image-block input to upload to <project>/assets/. Filenames are sanitized server-side; collisions get a numeric suffix. PNG, JPEG, GIF, SVG, WebP, AVIF up to 25 MB.

Exports

One renderer, three targets.

FormatCommandNotes
PDFverso build --format pdf [--path <id>]Headless Chromium. Default action of the editor's Export button.
HTMLverso build --format html [--no-inline-images]Single self-contained .html per path. Layout CSS, every slide, scale-to-fit stage, keyboard nav, hash sync. Images base64-inlined by default.
PNGverso build --format pngPer-slide screenshots at 1920x1080.

Per-path export

Pass --path <id> to render a single audience. Without it, every path is exported as its own file.

Speaker mode

Append ?mode=speaker to the viewer URL for a two-panel view: current slide, presenter notes, next-slide preview, and a stopwatch (click to reset).

Laser pointer

In present or speaker mode, click and drag anywhere on the slide to paint a fading red trail. Useful when screen-sharing. No toggle, no setup. Disabled in the editor's preview iframe.

Keyboard navigation

Arrows, PageUp / PageDown, Space, Home, End. Hash in the URL stays in sync so a refresh keeps you on the current slide.

Extending Verso

Drop a verso.config.ts at the project root to register custom layouts and components.

// verso.config.ts
import { defineConfig } from '@starside-io/verso-runtime'
import myCardWall from './layouts/card-wall'
import myMermaid from './components/mermaid'

export default defineConfig({
  layouts: { 'card-wall': myCardWall },
  components: { mermaid: myMermaid }
})

A layout receives a LayoutContext (resolved colors, the slide, the deck outline, the active path). A component is a function that takes a block and returns HTML.

Project styles

Drop a styles.css or verso.css at the project root and the viewer loads it automatically. Useful for new .verso-transition-<name> keyframes or theme tweaks.

Where the code lives

  • packages/schema/ · Zod schemas (Manifest, Slide, ContentBlock, Theme)
  • packages/core/ · pure helpers (color resolution, path filtering, content flattening)
  • packages/runtime/ · renderDeck, mount, layout / component / theme registries
  • packages/layouts/ · the 17 built-in layouts plus styles.css
  • packages/themes/ · the six built-in themes
  • packages/build/ · headless-Chromium PDF / HTML / PNG export
