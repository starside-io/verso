JSON in.
A presentation out.

Every slide is a plain JSON file. Diffable in git, scriptable from the CLI, generatable from an LLM. The visual editor is a UI on top of the same files.

Design choices

A presentation system, not a slideshow tool.

Treat the deck like a codebase: plain text, validated by a schema, composed of small reusable parts.

Zod-validated schema

Every write goes through Slide, Manifest, and ContentBlock parsers. Errors surface inline. Bad JSON never reaches disk.

Recursive blocks

card and panel hold their own content[]. Compose any block tree, with per-block path_include and style_overrides.

Pure-function renderer

renderDeck(manifest, slides, ctx) returns a string. Run it in Node, in a worker, at build time, anywhere.

The schema

Three small types. Zero magic.

A deck is a manifest plus a folder of slide files. A slide is a list of blocks. A block is a tagged union. That's the whole model.

deck.json

The manifest is the spine.

Paths are first-class. slide_order is the source of truth for sequence (you can also add string section markers).

{
  "title": "Q4 readout",
  "theme": "verso-slate",
  "paths": {
    "sales": { "label": "Sales", "color": "#5eead4" },
    "engineering": { "label": "Engineering", "color": "#a78bfa" }
  },
  "slide_order": [
    "01-cover", "02-context", "03-arch", "04-pricing", "05-closing"
  ],
  "variables": { "author": "M. Burlet" }
}
slides/02.json

Blocks are a tagged union.

Each block has a type. card and panel nest. Every block can carry path_include / path_exclude and style_overrides.

{
  "layout": "image-left",
  "content": [
    { "type": "image", "src": "assets/product.jpg" },
    { "type": "heading", "text": "Key features" },
    { "type": "bullets", "items": [
      "Single source of truth",
      "Audience branching"
    ]}
  ]
}
style_overrides cascade

Theme → deck → slide → block.

Each level emits CSS variables. Block-level wins. accent → secondary, surface/muted derived via color-mix().

{
  "id": "ba-deep-dive",
  "layout": "two-col",
  "style_overrides": {
    "primary": "#D95B4A",
    "background": "#1C1C1C"
  },
  "content": [{
    "type": "bullets",
    "style_overrides": { "secondary": "#FFD166" },
    "items": ["As-is mapping", "Gap analysis"]
  }]
}
Modular by design

Layouts, blocks, themes, plugins. All registries.

17

Layouts

Bundled set spans cover, agenda, two/three-col, image-left/right, hero, full-image, stats, big-number, quote, timeline. Register your own via verso.config.ts.

14

Block types

heading, text, bullets, image, code, quote, callout, badge, divider, accent-bar, embed, card, panel, custom. card/panel recurse.

16

Transitions

fade, slide, zoom, flip, iris, wipe, pop, blur, tilt. Free-form string at the schema, so you can register your own keyframes.

6

Themes

Three semantic roles (primary, secondary, classic) plus optional background/foreground. JSON file or inline.

⤓

Build targets

verso build --format pdf|html|png. PDF and PNG via headless Chromium, HTML inlines images for a single-file share.

{ }

Variables

{{date}}, {{time}}, {{deckTitle}}, {{pathId}} built-in. Custom keys live in manifest.variables.

AI-friendly by accident

If an LLM can write JSON, it can write a deck.

Every authoring instruction maps to a JSON operation. No proprietary file format, no XML archaeology, no binary blobs to unpack. 

// "Add a slide after intro-2 for the PM path"
{
  "id": "pm-overview",
  "layout": "two-col",
  "path_include": ["pm"],
  "content": [ /* … */ ]
}

// "Make this slide feel warmer"
{ "style_overrides": { "primary": "#C0622F" } }

// Markdown outline import: scaffolds one slide per heading
verso new slide --from outline.md
