Why Verso

A presentation system, not a slideshow tool.

Verso treats your deck like a codebase: plain JSON, diffable in git, versionable, and authored by humans or LLMs.

Branch by audience

Tag any slide or block with path_include / path_exclude. One source, multiple decks, zero copy-paste.

JSON-canonical

Every slide is a plain JSON file. Diffable, scriptable, AI-writable. The visual editor is a UI on top of the same files.

Export anywhere

PDF for print, single-file HTML for sharing, PNG for embedding. All driven by the same renderer, all pixel-identical.

By example

What you write. What you ship.

Each pair: the JSON on one side, the rendered slide on the other. Same source, no surprises.

layout: cover

A cover that doesn't apologize.

Slide-level title and header drive the layout. No template engine, no theme-specific HTML, just JSON.

{
  "id": "04-cover",
  "header": "Q4 2026 readout",
  "title": "One deck. Many audiences.",
  "layout": "cover"
}
layout: image-left

Mix any blocks. Split into columns.

Pure block list, layout picks the split. Images, bullets, callouts, code: they all compose without configuration.

{
  "layout": "image-left",
  "content": [
    { "type": "image",
      "src": "assets/product.jpg",
      "alt": "Product screenshot" },
    { "type": "heading", "text": "Key features" },
    { "type": "bullets", "items": [
      "Single source of truth",
      "Audience branching",
      "PDF, HTML, PNG export"
    ]}
  ]
}
layout: stats

Cards become a metric row.

The stats layout takes any number of cards and stretches them across the slide. Each card carries a tone that picks up from the active theme.

{
  "layout": "stats",
  "content": [
    { "type": "card", "tone": "primary",
      "content": [
        { "type": "heading", "level": 1, "text": "12.4K" },
        { "type": "text", "text": "Active decks shipped" }
      ] },
    /* + two more cards for +34% and 97% */
  ]
}
layout: quote

A single block, the right layout.

Pick the layout that says what you mean. The same quote block reads differently under content vs quote layout.

{
  "layout": "quote",
  "content": [{
    "type": "quote",
    "text": "We used to maintain three decks…",
    "attribution": "VP Marketing, mid-size SaaS"
  }]
}
Path branching

One source. Three audiences.

Tag any slide or block with path_include. The editor's Paths view shows the whole deck as a timeline: shared cards on the spine, audience-specific cards on branched lanes, and a colored line threading through every path.

deck.json  ·  6 slides  ·  3 paths
Sales Engineering Executive
Sales ?path=sales

Pitches branding + audience-targeted variants. The exec and engineering blocks are filtered out.

Engineering ?path=engineering

Surfaces the JSON schema, plugin registry, and CI pipeline story. Sales and exec blocks drop out.

Executive ?path=exec

Single source of truth, audit trail, vendor neutrality. Sales and engineering details disappear.

// slides/05-paths.json
{
  "layout": "two-col",
  "content": [
    /* shared blocks have no path filter and render everywhere */
    { "type": "heading", "text": "Included" },
    { "type": "bullets", "items": [ /* core features */ ]},

    /* path_include tags scope a block to one audience */
    { "type": "bullets", "path_include": ["sales"], "items": [ /* … */ ]},
    { "type": "bullets", "path_include": ["engineering"], "items": [ /* … */ ]},
    { "type": "bullets", "path_include": ["exec"], "items": [ /* … */ ]}
  ]
}

30-second quickstart

Install, scaffold, edit.

No accounts, no cloud, no proprietary file format. It's npm install, edit JSON, build.

1. Install the CLI

npm install -g @starside-io/verso-cli

2. Scaffold a deck from a template

verso init my-deck --template layouts-gallery
cd my-deck

3. Open the visual editor or edit JSON directly

verso edit            # full editor with live preview
verso dev             # just the viewer, watch JSON

4. Export when you're done

verso build --format pdf
verso build --format html   # single self-contained file
